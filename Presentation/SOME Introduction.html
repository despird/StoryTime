Introduction

SOME (Simple Object Model Edit) is a "language" for designing object-oriented applications. It aims to be a simple alternative for UML (Unified Modeling Language).

SOME vs UML

Like UML, SOME helps you design applications in an object-oriented way. It appears like a simple UML without diagrams. SOME has implemented two UML model types: Class model (SOME Roles) and Sequence model (SOME Stories). On the other hand, as a "language", SOME has a complete suite of language specifications.

Why SOME? (Benefits)

Simplicity
SOME can be regarded as a type of object-oriented pseudo-code, learning SOME is very easy if you are familiar with any other popular programming languages like C# or C++, Java. Unlike most languages, SOME does not have any keywords, instead, SOME uses some non-letter tokens such as '.', '->' or concatenating pre-defined prefix into identifier names to be part of a modifier. Please refer to the examples below(c#):

 "CSample" is interpreted as  "class Sample"
 "o_Method()" is interpreted as "public override void Method()", the capital M indicates this should be 'public'.
 "int r_Length" is interpreted as a definition of a read-only Property:
private int  _length; 
public int Length
{            
   get{return _length};
} 
"CSample s<CSubSample>.();" is interpreted as "CSample s = new CSubSample();".
 "obj;" is interpreted as "return obj;".

Readability
Unlike any popular OO language, which sacrifices readability for maintainability, SOME Stories reflect actual software run-time events. Thanks to the simplicity, SOME is perfect as a design language which provides a very clear means of describing software systems.

Code as documentation
SOME is an alternative for diagram-based documentation. Some people find it very frustrating to use CASE tools especially when they actually haven't saved any effort. Drawing class diagrams and sequence diagrams usually uses more time than writing actual code. Code masters rarely use a tool to draw diagrams, they draw them with a real pen on paper or whiteboards, because this saves time. Back to constructional programming ages, they mostly used pseudo-codes. Today we still have to draw design diagrams because of two major reasons: for designer himself to better review the design; for peers to leave a helpful documentation.

SOME is simple, so without any doubts it saves time. It contains design models so it can be used as documentation itself, thanks to its readability. It can easily be translated into target source code so it can also be seen as the code itself. It is a pseudo-code standard for elite programmers.
